Mission

Build is the craft and delivery engine of The Agency. We turn ideas into shippable, customizable, worth-paying-for products — fast.

"Is it shippable, customizable, and worth paying for?"

Deployment Modes
Mode 01
Forward Deploy
You are assigned to a specific client engagement. Symph is fully remote — but Forward Deploy PEs are available for on-site client visits when needed (typically once a week or bi-weekly). You are Symph's engineer closest to the client.
  • Builds and delivers the client's product hands-on
  • Demos features and progress directly to the client
  • Represents The Agency in person
  • Translates client needs into technical decisions in real time
  • Owns the engineering relationship (trust, communication, delivery)
  • Geographic matching by default (MNL PEs → Luzon clients, Cebu PEs → Cebu clients — active pilot given Cebu client volume); cross-region deployment available for high-stakes visits; any region at scale
Profile: Strong builder + strong communicator. Can walk into a client's office, demo, field questions, handle feedback on the spot.
Mode 02
Applications
You build from HQ. Symph’s core deployment mode for internal products, venture-mode startup cycles, and client projects delivered remotely. The engine behind Symph’s own product portfolio.
  • Builds and ships products from base (client or internal)
  • Supports multiple projects when needed
  • Handles remote client delivery via calls and async
  • Builds pre-client MVPs and venture cycle products
  • Serves as flex pool when Forward Deploy needs backup
  • No geographic constraint
Goal: Build a team where everyone is capable of Forward Deploy. Applications is the starting mode for most PEs.

Forward Deploy Readiness
Moving from Applications to Forward Deploy isn't a promotion — it's a deployment decision. Either path requires deliberation with the Lead AI Product Engineer before deployment. The rubric informs the decision; it does not make the decision.

A PE who meets all Hard Gates but falls short on one or more Soft Gates can still be deployed forward. The expectation is they develop those areas in the engagement.
Path B
Demonstrated Performance (Fast Track)
Real-world proof overrides the rubric. If a PE has already done the work and a client has seen the output, they've demonstrated Forward Deploy capability regardless of rubric tier.
  • Built and shipped at least one end-to-end product or PoC using Aria
  • That product has been demo'ed to a client (by them or by someone else — the product itself passed the bar)
Note on the checklist: This criteria isn't about gatekeeping — it's about ensuring we're setting people up for success when they walk into a client's office. If someone is clearly ready, they're ready. The checklist is a guide, not a gate.

The Decision Process

  1. 01PE qualifies via Path A or Path B (or both)
  2. 02Deliberation with Lead AI Product Engineer — reviews rubric data, working observation, team/client fit
  3. 03If ready: assigned to next available Forward Deploy engagement
  4. 04If not yet ready: specific growth areas identified, PE continues in Applications with targeted stepping-stone opportunities (e.g., co-demo on a client call)
Cross-Current Deployment

Pulling someone INTO Build

When Build needs additional capacity and no Build PEs are available, people from other currents (especially ex-developers) are assessed against the Build rubric. Hard Gates only for immediate deployment. Full rubric identifies ramp-up gaps. Both current leads must deliberate before deployment, time-boxed with a review point.

Deploying a Build PE to another current

When a Build PE is needed in another current (supporting Growth with technical demos, helping Taste with engineering-informed reviews), the receiving current's rubric determines readiness. Build PE's existing competencies — especially Client Communication, Product Thinking, and Cross-Current Collaboration — will often translate directly.

Template for Other Currents
Other current leads can use this structure to define their own rubrics:
  1. Adopt the 2 General Competencies as-is (Knowledge Sharing, Cross-Current Collaboration)
  2. Define 6–8 current-specific competencies relevant to your work
  3. Use the same 4-tier scale (Unacceptable / Capable / Adoptive / Transformative)
  4. Set your target state (Adoptive recommended as baseline for all currents)
  5. Define any deployment or role-specific readiness indicators if applicable
  6. Include Clue lines and Example Quotes for each tier to make assessments practical and consistent
